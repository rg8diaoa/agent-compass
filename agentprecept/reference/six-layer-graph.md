# 多维图案例：图可以走多远

> AgentPrecept 的三层图适用于大多数项目。本案例展示高度复杂领域的图模型——当你的项目需要更细粒度的建模时，可以从三层扩展。

---

## 案例：复杂系统模拟引擎

一个多 Agent 协作的模拟引擎项目，需要对 46 个独立配置维度进行建模。问题：如何让 Agent 理解这些维度的关系，而不在 46 个开关中迷失？

解决方案：六层图模型。将 46 个维度按领域分组到 6 个图层中：

| 图层 | 领域 | 示例维度 |
|---|---|---|
| **拓扑层** | 系统基础结构 | 模块、子模块、入口、接口 |
| **组织层** | 实体与关系 | 角色、组织、权限、层级 |
| **资源层** | 数据与资产 | 数据集、存储、缓存、配置 |
| **知识层** | 信息与逻辑 | 规则、算法、业务逻辑、设计模式 |
| **行为层** | 交互与约定 | 协议、事件、工作流、约定 |
| **历史层** | 事件与因果 | 日志、变更记录、决策链、回溯路径 |

---

## 与三层图的对应

| 三层图 | 六层图 |
|---|---|
| **结构层 S** | 拓扑层 + 行为层（系统的骨架和交互规则） |
| **关系层 R** | 组织层 + 资源层 + 知识层（实体间的关联网络） |
| **演变层 E** | 历史层（变化链和因果追溯） |

三层图是通用基线。六层图是领域的特化——当你的项目有 30+ 独立配置单元时值得考虑。

---

## 可复用模式：目录即注册

六层图中的每个维度通过独立配置文件自注册：

```
config/
├── modules.yaml       ← 模块维度
├── permissions.yaml   ← 权限维度
├── datasets.yaml      ← 数据集维度
├── rules.yaml         ← 规则维度
├── workflows.yaml     ← 工作流维度
└── ...
```

核心系统不持有配置列表。新增维度 = 新建一个配置文件。禁用维度 = 删文件或设 `enabled: false`。

这是一个**可插拔架构**的极致形态——引擎完全不知道"有哪些维度"，只知道"怎么读配置"。

---

## 三个通用原则

### 1. 连接即存在

一个实体没有边等价于不存在。质量的度量不是"有多少配置"，而是"配置之间有多少关联"。

### 2. 渐进构建

复杂系统不是一次性配置完成的。分三步：

- **第一步**：骨架级——定义顶层模块和主要接口
- **第二步**：关联级——填充模块间的依赖和实体间的关系
- **第三步**：细节级——具体的数据模型、规则逻辑、历史记录

### 3. 信息与载体分离

同一个业务规则可能有多种实现（代码中的函数、文档中的描述、测试用例中的断言）。它们是独立节点——修改一个不影响另一个的完整性。

## 从案例到你的项目

- **代码项目**：拓扑层=包结构，组织层=权限体系，资源层=数据库，知识层=业务规则，行为层=API 协议，历史层=git log
- **知识库**：拓扑层=目录结构，组织层=作者/编辑者，资源层=引用文献，知识层=概念网络，行为层=更新规则，历史层=修订记录
- **项目管理**：拓扑层=里程碑，组织层=团队，资源层=预算/工具，知识层=决策记录，行为层=流程，历史层=周报/复盘

不需要六层——三层就够了。但当你的项目有 30+ 独立配置项时，细分到六层能让 Agent 更精确地理解每个配置的职责边界。
