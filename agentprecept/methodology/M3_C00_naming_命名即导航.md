# 02 — 命名即导航

> **AgentPrecept 落地**: `check-naming.py` + 审计维度 1（命名一致性）已自动化，详见文末。

> Agent 找文件靠名字。名字好，Agent 秒找到。名字乱，Agent 永远迷路。

---

## 为什么命名规范是第一道防线

Agent 不是人类。人类可以容忍 `final_v2_revised_new.md` 这种文件名。Agent 要靠模式匹配来定位文档和代码。

实战数据：41 份文档，统一采用 `L{Level}_{CAT}{NN}_{Slug}_{Title}.md` 格式。三轮审计中"命名一致性"维度始终零问题。命名的稳定性是体系的根基。

---

## 文档命名格式

```
L{Level}_{Category}{NN}_{Slug}_{Title}.md
```

| 字段 | 示例 | 规则 |
|---|---|---|
| `L{Level}` | L1 / L2 / L3 / L4 | 阅读层级：数字越小越早读 |
| `{Category}` | A-P 单字母 | 文档分类（见分类表） |
| `{NN}` | 01-99 两位数字 | 分类内序号，连续不跳跃 |
| `{Slug}` | kebab-case | 英文短标识，≤25 字符，用于 grep |
| `{Title}` | ≤8 字 | 精简标题——不是完整描述 |

**示例**：

```
L1_A01_quickstart_快速开始.md      ← 正确
L1_A02_naming-convention_命名规范.md ← 正确
L2_D01_architecture_架构设计.md      ← 正确

quickstart.md                       ← 错误：没有层级和分类
L1_A01_快速开始_v3_final.md         ← 错误：版本后缀、中文 slug、标题超长
L2_D01_graph_model.md               ← 错误：没有中文标题
```

版本不体现在文件名里。修订走 Git 追踪。

## 文档状态机

每个文档头部标注状态，Agent 据此判断可靠性：

| 状态 | 含义 | Agent 应该 |
|:--:|------|---|
| ⏳ 待撰写 | 骨架已建，内容为空 | 不引用 |
| 📝 撰写中 | 内容正在编写 | 谨慎引用 |
| 🔍 审查中 | 内容完成，等待审查 | 可引用，欢迎纠错 |
| ✅ 已锁定 | 审查通过 | 可安全引用，不可修改 |
| ❌ 已废弃 | 不再使用 | 不引用 |

## 文件模板

```markdown
# {标题}

> 分类: {CAT} | 层级: L{Level} | 编号: L{Level}_{CAT}{NN}
> 状态: 📝 撰写中 | 目标读者: {读者}
> 前置阅读: {前置文档}

---

## 内容
```

## 三个原则

1. **可 grep**：Slug 用英文 kebab-case，Agent 可以用文件搜索精确匹配
2. **可排序**：层级+分类+序号，任何文件管理器里顺序都是对的
3. **可预测**：Agent 看到 `L2_D04` 就知道这是"架构设计类第 4 份、第 2 层"——不需要打开文件

---

## 方法论 M 体系

方法论 `methodology/` 使用独立的命名体系 `M{层级}_{类别}{NN}_{Slug}_{Title}.md`：

| 字段 | 语义 |
|------|------|
| M | 方法论标识（区别于 docs/ 的 L 体系） |
| 层级 | 1-4 难度递进：1 入门 / 2 协作 / 3 工程 / 4 运维 |
| 类别 | A-D 主题：A 流程 / B 设计 / C 开发 / D 审计 |
| NN | 类别内编号 00+，新篇末尾追加，不引发其他段重编号 |

与 docs/ 的 `L{1-4}_{A-P}{NN}` 体系独立——两棵树，一条引用。详见 `methodology/INDEX.md`。
