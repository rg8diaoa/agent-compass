# 04 — 审计框架：Agent 产出的质量检查系统

> **AgentPrecept 落地**: `agentprecept audit --gate` 15 维自动化 + `basic-audit.py`，详见文末落地表。

> Agent 产出需要审。不是靠直觉。是靠一套可以跑在 Agent 身上的检查清单。

---

## 为什么 Agent 产出需要结构性审计

Agent 擅长生成，不擅长自我纠错。一个 Agent 写了 30 份文档，里面可能存在：

- 文件名不符合规范（Agent 下次找不到）
- 文档 A 引用文档 B，但 B 的 slug 改了
- 术语在不同文档中用不同词汇
- 设计决策追溯丢失

这些问题单靠人工审查成本很高。但可以自动化——Agent 本身就能跑审计。

---

## 七维审计框架

每个维度一种检查类型。可以逐个维度交给 Agent 执行，也可以作为 CI 检查的一部分。

| 维度 | 检查什么 | 典型问题 |
|:--:|------|------|
| **1. 命名一致性** | 文件名是否符合规范 | 缺少 L 前缀、slug 用了中文、标题超长 |
| **2. 交叉引用完整性** | 文档 A 提到的文件路径是否真实存在 | 引用了已移动的文档、slug 拼写错误 |
| **3. 术语一致性** | 同一概念在所有文档中是否用同一个词 | "用户模块" vs "用户服务" vs "UserService" |
| **4. 设计决策追溯** | 每条关键决策是否有来源记录 | 设计变更日志为空、没有记录为什么选这个方案 |
| **5. 覆盖率** | 所有计划文档是否都已完成 | 状态标记与内容不一致、已废弃文档未清理 |
| **6. 骨架残留** | TODO/FIXME/TBD 占位符是否清理 | 文档说"待后续完善"但已过时 6 个月 |
| **7. 内容一致性** | 跨文档内容是否矛盾 | 架构图说 5 个模块、实施计划说 6 个 |

---

## 审计不是一次性的

一个 41 文档的实战项目进行了**三轮审计**：

| 轮次 | 🔴 阻塞项 | 🟡 建议项 | 变化 |
|:--:|:--:|:--:|------|
| r1 | 0 | 12 | 初建——命名和引用问题 |
| r2 | 2 | 12 | 重写核心设计后出现 8 个断链 |
| **r3** | **0** | **0** | 全部修复，体系稳定 |

每一轮不是在找"之前没发现的错"——是在验证**修改后是否产生了新的不一致**。

---

## 如何跑审计

1. 把这份框架交给 Agent
2. 让 Agent 逐维度扫描你的文档目录
3. Agent 输出审计报告：维度 → 发现 → 修复建议
4. 修复后跑第二轮

Agent 不需要理解你的项目——它只需要比较文件名、检查链接、扫描术语。这就是为什么审计是 Agent 擅长的任务。

## 扩展维度（真实项目中后续新增）

在原七维基础上，以下维度在 agentprecept 的自身审计实践中被验证有效：

### 维度 8: 狗粮审计

**项目自身是否遵循自己所倡导的方法论？**

- 项目是否有 INDEX.md（仓库导航）？
- 项目是否有 project-graph.yaml？
- 项目是否有 HANDOFF.md？
- 项目是否有设计依据（L4_O01）？
- 如果没有——你推荐别人用，自己为什么不用？

### 维度 9: 用户旅程审计

**从发现项目到用起来，每个环节是否有摩擦？**

- README 能否让人 30 秒理解"这是什么"？
- 有没有差异化对比（用 vs 不用）降低决策门槛？
- 有没有"已有项目接入"指南（不是所有用户都是新项目）？
- 有没有多路径（快速/深入/完整）满足不同用户？

### 维度 10: 编号连续性审计

**分类内模板编号是否从 01 开始连续？**

- 每个分类的 NN 是否从 01 开始？
- 中间是否有跳跃（如 E01→E03，跳过 E02）？
- 跳跃是刻意的还是遗漏？

## 延伸维度（审美与体验）

### 维度 11: 体验审计

**项目给人什么感觉？不是功能性——是阅读体验。**

- 是否有 300+ 行的文档巨墙（需要拆分）？
- 代码示例是否多、是否标注了语言（\`\`\`python vs \`\`\`）？
- 表格是否有对齐问题（长文本把列撑爆）？
- 目录是否有"分层感"——每个目录有 INDEX.md，不是文件堆？
- 每页打开后能否 5 秒判断"这页讲什么"？
- 是否有复制粘贴就能跑的示例（prompt/code block）？

### 维度 12: 定位审计

**项目是否让人一眼知道"这是干什么的、不是干什么的"？**

- README 第一段是否回答了"是什么、给谁用、解决什么问题"？
- 是否有"这不是什么"的边界说明（防止错误预期）？
- 与竞品/替代方案的差异是否说清楚了？
- 定位是否贯穿始终（README/INDEX/方法论开口一致）？
- 实战验证数据是否存在且可信？

## 进阶维度（工程健壮与社区就绪）

### 维度 13: 复用性与可移植性

**项目的各个部分能否独立工作？换个环境还能不能用？**

- 模板能否直接复制到新项目——无需手动改路径/占位符？
- 方法论是否绑定特定技术栈（Python only → Node 用户怎么办）？
- 能否只用一部分（只取 project-graph.yaml，不建完整文档体系）？
- 跨工具兼容性——AGENTS.md 在 Claude Code / Cursor / CodeWhale 中都能解析？
- 依赖外部工具的程度——是否需要安装额外软件才能用？

> **当前实现状态（2026-06-08）**：basic-audit.py 15 维自动化（--gate）+ 4 维自选清单。自动化覆盖：命名/断链/编号/骨架/图格式/追溯/覆盖率/狗粮（1-8）+ README声明/分支策略/commit粒度/术语表/数字矛盾/文档巨墙（9-15）。框架中维度 3（术语一致）/7（内容一致）/9（用户旅程）/11（体验）/12（定位）/13（复用）/14（社区）的语义部分因无法代码化，以自选清单在审计报告末尾提示，不纳入 FAIL。

### 维度 14: 社区就绪度

**项目准备好被陌生人使用和贡献了吗？**

- LICENSE / CONTRIBUTING / CODE_OF_CONDUCT 三件套齐全？
- Issue 模板是否覆盖 bug 和 feature？
- PR 模板是否有检查清单？
- 版本号是否规范（semver）？
- CHANGELOG 是否可读？
- 是否有反馈渠道（Discussion / Issue）？

## 防偷懒条款

审计时 Agent 最常见的偷懒行为：

| 偷懒行为 | 检测方法 | 硬规则 |
|------|------|------|
| "之前确认过，不用再查" | 要求每轮审计输出新的 shell 命令结果 | 审计前必须运行脚本 |
| "应该没问题" — 凭记忆判断 | 要求对照磁盘/YAML 实际数据 | 只信工具输出，不信记忆 |
| 工具失败后跳过该维度 | 审计报告维度数 ≠ 15 → 偷懒 | 失败维度标注 [无法验证] |
| 用之前会话的数据充数 | 文件修改时间戳对比 | 审计结果附带时间戳 |

## 一句话

**审计框架让 Agent 自己审查自己的产出——在你看到问题之前。**
