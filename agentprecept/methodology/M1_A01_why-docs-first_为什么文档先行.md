# 01 — 为什么文档优先

> **AgentPrecept 落地**: project-graph + 设计文档前置检查 + 15 维审计已全部工程化，详见文末。

> Code Agent 不是人类。它不会"大致记得"项目结构——它需要你告诉它。

---

## 问题的根源

给 Agent 一个 prompt，它能写出漂亮的代码。但项目一大，Agent 就开始迷失：

- 改了一个模块，不知道下游谁会被影响
- 写了新功能，命名风格和已有代码不一致
- 修了一个 bug，下个会话 Agent 又写出来了

根源不是 Agent 能力不足。根源是**Agent 没有项目的心智模型**。人类靠数月浸泡建立这种模型；Agent 只有你给它的上下文。

---

## 文档是 Agent 的上下文接口

给 Agent 一份好的文档，效果立竿见影。不是给一大堆。是给**对的那一份**。

实战验证：一个包含 41 份文档和 46 个独立配置维度的多 Agent 协作项目，Agent 拿到结构化的文档体系后，产出的一致性从"每次都像不同人写的"变成"跨三个会话能无缝衔接"。三轮审计从 12 个问题收敛到 0，外部评审 9.0/10。

---

## 三个层次，不是一堆 .md

| 层次 | 内容 | Agent 什么时候读 |
|---|---|---|
| **L1 蓝图** | 项目是什么、怎么跑起来、命名规则、术语 | 每次会话开始时 |
| **L2 核心** | 架构设计、数据模型、API 契约 | 做技术决策时 |
| **L3 实施** | 测试策略、验收标准、用户手册 | 写代码和验证时 |
| **L4 追溯** | 设计变更日志、设计依据、运维手册 | 维护和迁移时 |

Agent 不需要一次读完所有。它按任务按需加载——就像你不会一个上午读完所有 RFC。

---

## 四份"一等公民"文档

不管你用什么文档体系，这四份必不可少：

| 文档 | 角色 | 缺了会怎样 |
|---|---|---|
| **INDEX.md** | 目录骨架 | Agent 不知道有哪些文档 |
| **命名规范** | 文件/变量/类的命名宪法 | Agent 产出风格不一致 |
| **术语表** | 所有专用词的定义 | Agent 对同一概念用不同词 |
| **HANDOFF.md** | 会话交接记录 | 下个 Agent 不知道上一步做了什么 |

---

## 这不是额外的负担

`AGENTS.md` 就是这个方法论的入口。把它放在你的项目根目录，Agent 自动读取。

你不需要从零写 41 份文档。从一份 INDEX.md 和一页命名规范开始。项目长到 Agent 开始犯错的时候，再加一层。

**文档的代价是线性的。没有文档的代价是指数的。**

## 常见借口（反模式对照）

| 借口 | 为什么行不通 |
|------|-------------|
| "文档会过时" | agentprecept sync 从代码自动更新 project-graph，文档和代码同步 |
| "小项目不需要" | 1 个模块时无所谓——3 个模块时 Agent 开始迷路。文档是保险，不是装修 |
| "Agent 自己会读代码" | 读代码重建心智模型需要 5-10 轮对话。project-graph 30 秒 |
| "先写代码再补文档" | "先上车再买票"模式。补文档永远排在下一个 sprint 后面 |
| "设计文档限制 Agent 发挥" | 设计文档是护栏不是牢笼。有护栏的高速公路比野地跑得快 |
