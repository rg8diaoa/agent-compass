# M2_B02 — 非技术审 Agent 产出

> 方法论体系: M{层级}_{类别}{NN} | 层级: 2(协作) | 类别: B(设计)
> 状态: 📝撰写中 | 目标读者: 非技术用户

> 你不懂代码。但你能判断 Agent 做得对不对——这一篇教你怎么不看代码也能审产出。

---

## 不看代码看什么

| 看什么 | 怎么验证 |
|--------|----------|
| 功能完整性 | 对照你最初的需求，逐条试一遍 |
| 操作流畅度 | 点完所有按钮，有没有报错或空白页 |
| 数据正确性 | 输入数据 → 查输出结果是否对 |
| 错误处理 | 输入明显错误的数据，看会不会崩 |

**不看**：代码风格、变量命名、架构设计——那是 Agent 之间交接的事。

## 对照需求清单逐条勾选

Agent 出 checklist 后，把它当验收清单：

```
用户需求: 日记应用
Agent checklist:
  [x] 新建日记
  [x] 按日期浏览
  [ ] 全文搜索 ← 你没试这个
  [x] 删除日记
```

少勾一项 → 告诉 Agent "全文搜索我试了没反应"。

## 跑起来 > 代码好

```
❌ 审查方式: 打开源码文件，看代码逻辑
✅ 审查方式: 打开应用，点完所有功能

❌ 反馈: "login.py 的 try/except 结构可能需要优化"
✅ 反馈: "登录页输错密码没有提示，直接就刷新了"
```

## 怎么给 Agent 反馈

| 你发现的 | 怎么说 |
|----------|--------|
| 功能没了 | "我点了 XX 按钮，什么都没发生" |
| 结果不对 | "输入 100，显示的结果是 200，差了 1 倍" |
| 样子不对 | "这个按钮在手机上被截掉了一半" |
| 不知道什么意思 | "这个词我不知道什么意思，能换成 XX 吗" |

不说 "代码有问题"——说 "功能有问题"。Agent 自己会定位代码。

## 4 项人类审 Agent

| 检查项 | 怎么看 |
|--------|--------|
| □ 模块 > 3 | 打开 project-graph.yaml，数有几个大模块 |
| □ 有职责描述 | 每个模块下面有没有一行中文说明 |
| □ L4_O01 有依据 | 有没有记录"为什么这样设计" |
| □ 图已更新 | 最新一条日期是不是今天的 |

4 项全 □ → 及格。少 1 项 → 告诉 Agent "补一下这里"。

## 与 AgentPrecept 的衔接

- AGENTS.md 规则：Agent 必须标注 [NEEDS_HUMAN_REVIEW] 等你确认
- 4 项审 Agent 清单：`agentprecept audit --gate` 自动检查 15 维
- handoff 交接：换 Agent 时新 Agent 自动读 HANDOFF 知道做到哪了
