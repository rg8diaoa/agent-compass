# 05 — 会话交接：Agent 间的上下文传递

> **AgentPrecept 落地**: `HANDOFF.md` 模板 + `handoff_read` MCP tool + AGENTS.md 规则，详见文末。

> Agent 每次会话都是新的。HANDOFF.md 是上一个 Agent 给下一个 Agent 留的纸条。

---

## 问题

你今天和 Agent 一起调试了一下午。修了 3 个 bug、重构了一个模块、写了 2 页设计文档。

明天你打开新的会话。Agent 不知道今天发生过什么。

除非你留了纸条。

---

## HANDOFF.md 模式

这是实战项目中每轮会话结束时 Agent 全量重写的交接文档。内容如下：

```markdown
# 工作交接

> 最后更新: YYYY-MM-DD
> 当前状态: [简短的一句话状态描述]

---

## 当前状态

- 哪些做完了，哪些做到一半
- 哪些被阻塞，为什么

---

## 本会话做了什么

- 逐条列出具体完成的事项

---

## 下一步

- 下一个会话应该从哪开始（具体的一步，不是方向性的一句话）

---

## 关键决策

- 本会话做出的有长期影响的选择
- 如果有设计依据文档，指向对应条目
```

---

## 规则

1. **全量重写**，不增量修改。每次会话结束，HANDOFF.md 完全替换为一页新内容。历史版本由 Git 保留（`git log -- HANDOFF.md`）
2. **下一个 Agent 的第一项任务**是读 HANDOFF.md——在 `AGENTS.md` 或索引中声明这一点
3. **只写事实**，不写猜测。`"测试失败原因待排查"` 而不是 `"可能是竞态条件"`
4. **下一步必须具体**。不是 `"继续开发"`，而是 `"运行 test_auth.py 中的 3 个失败用例，从第 47 行的 mock 开始排查"`

---

## 和设计依据 / 变更日志的关系

| 文档 | 粒度 | 频率 | 读的人 |
|---|---|---|---|
| **HANDOFF.md** | 一次会话 | 每次会话 | 下一个 Agent |
| **设计依据** | 一条决策 | 决策后 | 所有未来 Agent |
| **变更日志** | 一次发布/审查 | 版本级 | 维护者 |

HANDOFF 是即时便签，设计依据是永久档案，变更日志是版本记录。

---

## 和 Git commit message 的区别

Git 知道代码变了什么。它不知道：

- 为什么这个方案比另一个好（但还没到写设计依据的程度）
- 哪个 bug 的根因还没找到
- 下一个开发者应该从哪里接手

HANDOFF 填补的是 commit message 和代码之间的空白。

---

## 一句话

**如果每个 Agent 会话结束时都能留下一页纸条，Agent 协作就不再需要人类当"记忆中转站"。**

## AgentPrecept 工程化落地

| 理念 | 实现 | 使用方式 |
|------|------|----------|
| 交接模板 | `HANDOFF.md` 标准模板 | `agentprecept init` 自动复制 |
| 交接读取 | `handoff_read` MCP tool | `handoff_read()` |
| 规则触发 | AGENTS.md 会话结束信号自动重写 | 每次会话结束自动执行 |
