# 03 — 设计依据：Agent 不推翻已验证的决策

> **AgentPrecept 落地**: `L4_O01` 模板 + `decision_search` MCP tool + 审计维度 6（设计追溯），详见文末。

> 每条设计决策都应该回答一个问题：我们试过什么？为什么这个更好？

---

## Agent 的"重蹈覆辙"问题

你告诉 Agent 修一个 bug。Agent 看了代码，说"这个架构不对，应该重构"。它提出一个方案——和三个月前团队讨论过然后否决的方案一模一样。

Agent 不知道那段讨论发生过。它只看到代码。它不理解**为什么**。

---

## 设计依据文档

设计依据文档不描述"是什么"——架构文档已经做了那件事。设计依据只回答"为什么"：

```
设计决策：为什么用 Redis 而不是数据库做缓存？

来源：2025-03-15 性能测试
证据：数据库缓存 QPS 1200，Redis QPS 8500
决策人：后端团队
替代方案：memcached（API 不够丰富）、本地 cache（多实例同步问题）
```

---

## 模板

每行设计决策记录为一行表格：

| 决策 | 来源 | 证据 |
|---|---|---|
| 为什么 X 不用 Y | 哪次测试/哪个教训 | 数据或具体事件 |
| 为什么不选 Z | 哪次讨论/哪个评审 | Z 被否决的原因 |

实战案例：一个多 Agent 协作项目在设计依据文档中记录了 30+ 条设计决策，每条都追溯到前代版本的具体教训——"这个架构模式来自上版本的审核链崩溃"，"这个阈值来自上一个项目的线上 bug"。

Agent 读到这些之后，不会再把"已经被证明失败的模式"重新提出。

---

## 什么时候写

不是在项目开始前写完所有设计依据。这样写不出来——很多决策还没发生。

**在每个关键决策做出后立即写一行**。它应该是 3 分钟的动作，不是 3 小时的仪式。

---

## 和架构文档的关系

| 架构文档 | 设计依据 |
|---|---|
| 描述当前设计 | 解释为什么这样设计 |
| Agent 读它实现功能 | Agent 读它避免推翻决策 |
| 会随时间演进 | 只追加，不修改 |
| 位于 L2（核心） | 位于 L4（追溯） |

---

## 一句话

**不给 Agent 设计依据，它会在每次会话中重新发明轮子——而且通常是对方轮子。**
