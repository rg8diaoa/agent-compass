# 14 — 生产就绪：Agent 产出及格线

> **AgentPrecept 落地**: `agentprecept audit --gate` + CI Gate 15 维全自动门禁，详见文末。

> 每个阶段做到什么程度才算合格？不是"看起来可以"——是硬门槛。本指南同时服务两类读者：
> - **Agent**：自我检查"我的产出够格提交吗"
> - **人类**：评判"Agent 的产出应该通过哪些条件"

---

## 8 阶段退出标准

### 阶段 1：想法 → 需求

| # | 标准 | 验证 |
|---|------|:--:|
| 1 | 用户故事 ≥ 3 个 | ✅ 脚本 |
| 2 | 每个故事有验收条件（非空） | ⚠️ Agent 自检 |
| 3 | 新术语出现在 L1_B01 中 | ⚠️ Agent 自检 |
| 4 | 需求与已有功能不矛盾（查 L4_O01） | ❌ 人类 |

> ✅ = 脚本可验证；⚠️ = Agent 可以说"我做了"但人类应抽查；❌ = 必须人类审

**人类评判**：□ 看完 3 个用户故事，确认这就是你要做的

---

### 阶段 2：设计 → 架构

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | project-graph 结构层 ≥ 3 个模块 | ✅ |
| 2 | L4_O01 中 ≥ 1 条技术选型依据 | ✅ |
| 3 | 每条技术选型对比过 ≥ 1 个替代方案 | ❌ |
| 4 | 无循环依赖（madge --circular 通过） | ✅ |

**人类评判**：□ 读 L4_O01 的技术选型行——能理解"为什么选 X 不选 Y"吗？

### 阶段 4：开发

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | 所有公开函数有类型注解（pyright/mypy --strict 通过） | ✅ |

### 阶段 5：测试

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | 覆盖率 ≥ 80%（pytest --cov-fail-under=80） | ✅ |
| 2 | 每个 API 端点 ≥ 1 个正常 + 1 个异常用例 | ❌ |
| 3 | project-graph 中有 tests 边（每个测试文件 → 对应模块） | ✅ |

**人类评判**：□ 看 test 文件——每个文件有 "TC-{MODULE}-01" 和 "TC-{MODULE}-02"

---

### 阶段 6：审计

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | 15 维自动化审计通过（agentprecept audit --gate） | ✅（exit 0） |
| 2 | 0 🔴 阻塞项 | ❌ |
| 3 | 审计报告已生成（AUDIT_REPORT.md） | ✅ |

**人类评判**：□ 看审计报告最后一行——是 "0 阻塞项" 吗？

---

### 阶段 7：修复

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | 影响范围内全量测试通过 | ✅ |
| 2 | L4_O01 中记录了修复类型（点状/系统级） | ❌ |
| 3 | 如果系统级 → L4_O01 有根本原因记录 | ❌ |

**人类评判**：□ 修复 commit 的 message 说清楚了"为什么这样修"

---

### 阶段 8：维护 → 发布就绪

| # | 标准 | 自动检查 |
|---|------|:--:|
| 1 | Dockerfile 或部署脚本存在 | ✅ |
| 2 | .gitignore 存在 | ✅ |
| 3 | 健康检查端点存在（GET /health → 200） | ✅ |
| 4 | semver 版本号打在 git tag 上（`git tag -l` 非空） | ✅ |
| 5 | CHANGELOG 有本次发布条目 | ✅ |
| 6 | HANDOFF 状态 = [IN_PROGRESS]（新周期开始） | ✅ |

**人类评判**：□ `git tag -l` — 能看到版本号吗？

## Agent 自我检查规则

```
提交前自动跑:
  □ 阶段 4 退出标准 #1-#5（安全检查）
  □ 阶段 5 退出标准 #1（覆盖率）
  □ 如果改了 project-graph → 阶段 4 退出标准 #2

不通过 → 不提交。在 HANDOFF 中标注哪个标准未通过。
```

## 人类评判 Agent 产出（4 项及格线）

放到 `examples/first-run.md` 末尾。用户不懂架构也能审：

```
Agent 提交了架构设计 → 你对照:
  □ 模块划分表有 > 3 个模块
  □ 每个模块有 1 行职责描述
  □ L4_O01 中有对应的技术选型依据
  □ project-graph 已经更新

4 项全 □ → 及格。少 1 项 → 告诉 Agent "补一下 X"
```
